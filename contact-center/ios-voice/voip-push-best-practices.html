iOS VoIP Push Integration
Best Practices

A complete guide to integrating PushKit, CallKit, and the Vonage Voice Client SDK in a production iOS application — from token registration to call teardown.

Overview

Why PushKit + CallKit?

VoIP push notifications delivered via PushKit wake the app regardless of its state (foreground, background, or killed). Unlike standard APNs notifications, VoIP pushes have higher priority and are guaranteed to reach the device even in low-power modes.

CallKit is the mandatory companion: Apple requires that any app receiving VoIP pushes integrates with CallKit to display native call UI (lock screen, in-call screen, call log). An app that receives VoIP pushes without reporting them to CallKit is considered abusive — PushKit will silently stop delivering pushes to it.

The Vonage Voice Client SDK sits between the two: it decodes incoming push payloads into call invites, manages the WebRTC session, and provides the answer / reject / hangup verbs that close the loop.

High-level component responsibilities

Component Responsibility
PushKit (PKPushRegistry) Receive VoIP push tokens and incoming push payloads from APNs
Vonage SDK (VGVoiceClient) Register tokens, decode push payloads, manage WebRTC session, answer/reject/hangup
CallKit (CXProvider) Display native call UI, manage audio session, receive user actions (answer, end)
Setup

Project Configuration

1 — Xcode capabilities

Under your target's Signing & Capabilities, add:

  • Push Notifications — required for APNs registration.
  • Background Modes → check Voice over IP and Background processing.

Xcode automatically adds com.apple.developer.pushkit.unrestricted-voip to your .entitlements file when you enable the VoIP background mode.

2 — Push certificate

  • In the Apple Developer portal, create an Apple Push Notification service SSL (Sandbox & Production) certificate for your App ID.
  • Export it as a .p12 from Keychain Access and upload it to the Vonage API Dashboard under your application's push settings.
  • Certificates expire annually — track the expiry date. Expiry causes all VoIP pushes to fail silently with no error surfaced to your app.
Sandbox vs Production: Xcode debug builds use the sandbox APNs environment; TestFlight and App Store builds use production. Upload separate certificates for each environment and always set the isSandbox flag at compile time using #if DEBUG — never hardcode it.

3 — Frameworks

Add CallKit under Frameworks, Libraries, and Embedded Content in your target's General tab. Import as needed: 

import PushKit
import CallKit
import VonageClientSDKVoice
import AVFoundation

4 — SDK initialisation

Initialise VGVoiceClient once, early in the app lifecycle (e.g. in your AppDelegate or a singleton manager). Set its delegate before registering for push or creating sessions: 

let config = VGClientInitConfig(loggingLevel: .info)
let client = VGVoiceClient(config)
client.delegate = self
PushKit

PushKit Registration & Token Handling

Register for VoIP pushes

Create a PKPushRegistry and set its desired push types. Do this at app launch, before any call can arrive. 

class AppDelegate: UIResponder, UIApplicationDelegate {

    private let voipRegistry = PKPushRegistry(queue: nil)

    func application(_ application: UIApplication,
                     didFinishLaunchingWithOptions launchOptions: [UIApplication.LaunchOptionsKey: Any]?) -> Bool {

        // Request microphone permission early
        AVAudioSession.sharedInstance().requestRecordPermission { granted in
            print("Microphone permission: \(granted)")
        }

        voipRegistry.delegate = self
        voipRegistry.desiredPushTypes = [.voIP]

        return true
    }
}

PKPushRegistryDelegate

Implement the three delegate methods. The third — didReceiveIncomingPushWith — is the most critical: 

extension AppDelegate: PKPushRegistryDelegate {

    // Called when a new VoIP token is issued or rotated
    func pushRegistry(_ registry: PKPushRegistry,
                      didUpdate pushCredentials: PKPushCredentials,
                      for type: PKPushType) {
        guard type == .voIP else { return }
        // Store token; register with Vonage only when a session is active
        CallManager.shared.voipTokenDidUpdate(pushCredentials.token)
    }

    // Called when the token is invalidated (e.g. app reinstall)
    func pushRegistry(_ registry: PKPushRegistry,
                      didInvalidatePushTokenFor type: PKPushType) {
        CallManager.shared.unregisterDeviceToken()
    }

    // Called when an inbound VoIP push is received
    func pushRegistry(_ registry: PKPushRegistry,
                      didReceiveIncomingPushWith payload: PKPushPayload,
                      for type: PKPushType,
                      completion: @escaping () -> Void) {
        guard type == .voIP else { completion(); return }
        CallManager.shared.handleIncomingPush(payload: payload, completion: completion)
    }
}

Registering the token with Vonage

Register with client.registerVoipToken(_:isSandbox:) only when a Vonage session exists. Store the returned deviceId — you need it to unregister on logout. Do not call registerVoipToken on every session restore; once registered, the token remains valid until it rotates or the user logs out. 

func registerVoipToken(_ token: Data) {
    #if DEBUG
    let isSandbox = true     // Xcode / development → sandbox APNs
    #else
    let isSandbox = false    // TestFlight / App Store → production APNs
    #endif

    client.registerVoipToken(token, isSandbox: isSandbox) { [weak self] error, deviceId in
        if let error {
            print("❌ Token registration failed: \(error)")
            return
        }
        // Persist deviceId — needed for unregistration on logout
        UserDefaults.standard.set(deviceId, forKey: "vonage_device_id")
        print("✅ Token registered: \(deviceId ?? "")")
    }
}

func unregisterDeviceToken(completion: (() -> Void)? = nil) {
    guard let deviceId = UserDefaults.standard.string(forKey: "vonage_device_id") else {
        completion?(); return
    }
    client.unregisterDeviceTokens(byDeviceId: deviceId) { _ in
        UserDefaults.standard.removeObject(forKey: "vonage_device_id")
        completion?()
    }
}
Critical

Session Management & JWT Strategy

A Vonage session (client.createSession(jwt)) is required before the SDK can process call invites or perform call actions. When a VoIP push arrives and the app is killed or the session has expired, the session must be recreated. How fast this happens directly determines whether users can successfully answer calls.

The most common production failure

If you fetch a fresh JWT from your backend at push-receipt time, you are adding an uncached network round-trip — potentially 500 ms to 3 s on a mobile connection — before createSession can even start. The user sees the CallKit screen, taps Answer before the session is ready, and the call fails silently.

✗ Avoid

On push receipt: call your auth server → get fresh JWT → call createSession. Two serial network hops in a background execution window with poor signal.

✓ Recommended

At user login: persist the JWT (and a refresh token) locally. On push receipt: call createSession with the stored JWT directly. Zero backend calls on the fast path.

Session creation

func createSession(jwt: String,
                    onSuccess: ((String) -> Void)? = nil,
                    onError: ((Error) -> Void)? = nil) {
    client.createSession(jwt) { [weak self] error, sessionId in
        if let error { onError?(error); return }
        guard let sessionId else { return }

        self?.activeSessionId = sessionId
        // Persist token for future session restores
        UserDefaults.standard.set(jwt, forKey: "vonage_jwt")
        onSuccess?(sessionId)
    }
}

Session restoration on push receipt

Follow a priority chain — use the fastest available path:

Priority Condition Action Typical latency
1st Active session exists Return immediately — no work needed 0 ms
2nd Stored JWT is valid client.createSession(storedJWT) 100–400 ms
3rd JWT expired, refresh token available Refresh endpoint → new JWT → createSession 300–1000 ms
4th No credentials Log warning; push still processed — call cannot be answered 0 ms 
func restoreSessionIfNeeded() {
    guard activeSessionId == nil else {
        print("✅ Session already active")
        return
    }

    if let jwt = UserDefaults.standard.string(forKey: "vonage_jwt") {
        // Fast path — use stored JWT directly
        createSession(jwt: jwt)
    } else if let refreshToken = storedRefreshToken {
        // Slow path — exchange refresh token for a new JWT first
        refreshJWT(refreshToken) { [weak self] newJWT in
            self?.createSession(jwt: newJWT)
        }
    } else {
        print("⚠️ No credentials available for session restoration")
    }
}
Critical

Handling an Incoming VoIP Push

This is the most timing-sensitive piece of the integration. Two operations must happen from handleIncomingPush, and the order relationship between them is non-obvious.

The concurrent pattern (required)

PushKit delivers push — background budget begins
didReceiveIncomingPushWith is called. A background execution window opens. reportNewIncomingCall must be called promptly.
App validates push type & stores completion
Check VGVoiceClient.vonagePushType(). Store the PushKit completion block — it will be called only after reportNewIncomingCall has fired.
App fires session restore + push processing concurrently
restoreSessionIfNeeded() and client.processCallInvitePushData() are called without awaiting each other. Session restore is a background concern; it must never block push processing.
Vonage SDK decodes payload → fires didReceiveInviteForCall
processCallInvitePushData parses the push and triggers the VGVoiceClientDelegate callback with callId, caller, and channel type.
CallKit reportNewIncomingCall → then PushKit completion
Inside the delegate: call provider.reportNewIncomingCall. Inside its async callback, call the stored PushKit completion. This sequence is mandatory and must never be reversed.
Never await session restoration before calling processCallInvitePushData. Gating push processing behind createSession delays reportNewIncomingCall by a full network round-trip. Apple treats this as a failure to handle the push — PushKit silently stops delivering future pushes to the device.

Implementation

// Stored PushKit completion — held until CallKit reports the call
private var pendingPushCompletion: (() -> Void)?

func handleIncomingPush(payload: PKPushPayload, completion: @escaping () -> Void) {

    // 1. Validate: only handle Vonage incoming-call pushes
    let pushType = VGVoiceClient.vonagePushType(payload.dictionaryPayload)
    guard pushType == .incomingCall else {
        // Must still report to CallKit — failing to do so breaks future push delivery.
        // Report a call then immediately end it.
        let fakeUUID = UUID()
        callProvider.reportNewIncomingCall(with: fakeUUID, update: CXCallUpdate()) { _ in
            self.callProvider.reportCall(with: fakeUUID, endedAt: .now, reason: .failed)
            completion()
        }
        return
    }

    // 2. Store completion — will be called after reportNewIncomingCall
    pendingPushCompletion = completion

    // 3a. Restore session concurrently — non-blocking, runs in background
    restoreSessionIfNeeded()

    // 3b. Process the push immediately — triggers didReceiveInviteForCall
    client.processCallInvitePushData(payload.dictionaryPayload)
}
CallKit

CallKit Integration

CXProvider setup

Create CXProvider and CXCallController once and keep them for the lifetime of your manager. Do not recreate them on each call or push. 

class CallManager: NSObject {

    static let shared = CallManager()

    private let callProvider: CXProvider
    private let callController = CXCallController()

    private override init() {
        let config = CXProviderConfiguration()
        config.supportsVideo            = false
        config.maximumCallGroups        = 1
        config.maximumCallsPerCallGroup = 1
        // Use .phoneNumber only for E.164-formatted callers
        config.supportedHandleTypes     = [.generic]
        config.supportsDTMF             = true
        config.supportsHolding          = true

        callProvider = CXProvider(configuration: config)
        super.init()
        callProvider.setDelegate(self, queue: nil)
    }
}

Reporting an incoming call to CallKit

Called from didReceiveInviteForCall (see SDK Delegate section). The PushKit completion must be invoked inside the reportNewIncomingCall callback — never before it. 

func reportIncomingCall(callUUID: UUID,
                        caller: String,
                        type: VGVoiceChannelType,
                        pushCompletion: @escaping () -> Void) {
    let update = CXCallUpdate()
    update.remoteHandle  = CXHandle(type: type == .phone ? .phoneNumber : .generic,
                                     value: caller)
    update.hasVideo      = false
    update.supportsDTMF  = true
    update.supportsHolding = true
    update.localizedCallerName = caller

    callProvider.reportNewIncomingCall(with: callUUID, update: update) { error in
        if let error {
            print("❌ CallKit rejected incoming call: \(error)")
        } else {
            print("✅ CallKit notified of incoming call")
        }
        // ← This is the ONLY correct place to call the PushKit completion.
        //   Calling it here guarantees reportNewIncomingCall always fires first.
        pushCompletion()
    }
}

CXProviderDelegate — user actions

extension CallManager: CXProviderDelegate {

    func providerDidReset(_ provider: CXProvider) {
        // Called when CallKit resets (e.g. app backgrounded after a failed call).
        // Clean up any active call state here.
        activeCall = nil
    }

    // User tapped Answer
    func provider(_ provider: CXProvider, perform action: CXAnswerCallAction) {
        guard let callId = activeCallId else { action.fail(); return }

        client.answer(callId) { error in
            if error == nil {
                action.fulfill()
            } else {
                action.fail()
            }
        }
    }

    // User tapped End (covers both Reject while ringing and Hang up during call)
    func provider(_ provider: CXProvider, perform action: CXEndCallAction) {
        guard let callId = activeCallId else { action.fail(); return }

        let complete: (Error?) -> Void = { error in
            error == nil ? action.fulfill() : action.fail()
        }

        if isCallRinging {
            client.reject(callId, callback: complete)
        } else {
            client.hangup(callId, callback: complete)
        }
    }

    // CallKit activates audio — enable SDK audio here, not before
    func provider(_ provider: CXProvider, didActivate audioSession: AVAudioSession) {
        VGVoiceClient.enableAudio(audioSession)
    }

    // CallKit deactivates audio — disable SDK audio here
    func provider(_ provider: CXProvider, didDeactivate audioSession: AVAudioSession) {
        VGVoiceClient.disableAudio(audioSession)
    }

    // Mute toggle (e.g. from CarPlay or lock screen)
    func provider(_ provider: CXProvider, perform action: CXSetMutedCallAction) {
        guard let callId = activeCallId else { action.fail(); return }

        if action.isMuted {
            client.mute(callId) { _ in action.fulfill() }
        } else {
            client.unmute(callId) { _ in action.fulfill() }
        }
    }

    // Hold toggle
    func provider(_ provider: CXProvider, perform action: CXSetHeldCallAction) {
        guard let callId = activeCallId else { action.fail(); return }

        if action.isOnHold {
            client.enableEarmuff(callId) { _ in action.fulfill() }
        } else {
            client.disableEarmuff(callId) { _ in action.fulfill() }
        }
    }

    // DTMF (dialpad input from CarPlay or lock screen)
    func provider(_ provider: CXProvider, perform action: CXPlayDTMFCallAction) {
        guard let callId = activeCallId else { action.fail(); return }
        client.sendDTMF(callId, withDigits: action.digits) { _ in action.fulfill() }
    }
}
Audio session rule: always enable and disable Vonage audio exclusively inside didActivate / didDeactivate. Enabling audio before CallKit activates the session, or from a UI button handler, leads to audio routing bugs — especially with Bluetooth and CarPlay.
SDK Delegate

VGVoiceClientDelegate — Inbound Call Events

Incoming call invite

extension CallManager: VGVoiceClientDelegate {

    func voiceClient(_ client: VGVoiceClient,
                     didReceiveInviteForCall callId: VGCallId,
                     from caller: String,
                     with type: VGVoiceChannelType) {

        guard let callUUID = UUID(uuidString: callId) else { return }

        activeCallId  = callId
        activeCallUUID = callUUID
        isCallRinging = true

        // Report to CallKit, then — inside the callback — call the PushKit completion
        reportIncomingCall(callUUID: callUUID, caller: caller, type: type,
                           pushCompletion: pendingPushCompletion ?? {})
        pendingPushCompletion = nil
    }

    // Remote party hung up or the call timed out
    func voiceClient(_ client: VGVoiceClient,
                     didReceiveHangupForCall callId: VGCallId,
                     withQuality callQuality: VGRTCQuality,
                     reason: VGHangupReason) {

        guard let uuid = activeCallUUID else { return }

        let cxReason: CXCallEndedReason = switch reason {
            case .remoteReject:           .declinedElsewhere
            case .remoteHangup,
                 .localHangup:             .remoteEnded
            case .remoteNoAnswerTimeout:   .unanswered
            default:                      .failed
        }

        callProvider.reportCall(with: uuid, endedAt: .now, reason: cxReason)
        cleanUpCallState()
    }

    // Caller cancelled the invite before it was answered
    func voiceClient(_ client: VGVoiceClient,
                     didReceiveInviteCancelForCall callId: VGCallId,
                     with reason: VGVoiceInviteCancelReason) {

        guard let uuid = activeCallUUID else { return }

        let cxReason: CXCallEndedReason = switch reason {
            case .answeredElsewhere:   .answeredElsewhere
            case .rejectedElsewhere:   .declinedElsewhere
            case .remoteCancel:        .remoteEnded
            case .remoteTimeout:       .unanswered
            default:                   .failed
        }

        callProvider.reportCall(with: uuid, endedAt: .now, reason: cxReason)
        cleanUpCallState()
    }

    private func cleanUpCallState() {
        activeCallId   = nil
        activeCallUUID = nil
        isCallRinging  = false
    }
}
Always call callProvider.reportCall(with:endedAt:reason:) for every termination path — remote hangup, remote cancel, media timeout, local reject. Missing any of these leaves CallKit in an inconsistent state and can break the call log and future call reporting.
Outbound

Outbound Calls with CallKit

For outbound calls the flow is reversed: your app initiates the call via the SDK, then uses CXCallController to inform CallKit. Use reportOutgoingCall to track connection state. 

// 1. Initiate the call via the SDK
func startCall(to callee: String) {
    let callContext = ["callee": callee]

    client.serverCall(callContext) { [weak self] error, callId in
        guard let callId,
              let callUUID = UUID(uuidString: callId) else { return }

        self?.activeCallId   = callId
        self?.activeCallUUID = callUUID

        // 2. Notify CallKit so the in-call UI appears
        let handle = CXHandle(type: .generic, value: callee)
        let action = CXStartCallAction(call: callUUID, handle: handle)
        let transaction = CXTransaction(action: action)

        self?.callController.request(transaction) { [weak self] error in
            guard error == nil else { return }
            // Report that we've started connecting
            self?.callProvider.reportOutgoingCall(with: callUUID, startedConnectingAt: .now)
        }
    }
}

// 3. CXStartCallAction — CallKit confirming the transaction
func provider(_ provider: CXProvider, perform action: CXStartCallAction) {
    // Call was already started above; just fulfill
    action.fulfill()
}

// 4. When the remote side answers — fired via VGVoiceClientDelegate
func voiceClient(_ client: VGVoiceClient,
                 didReceiveLegStatusUpdateForCall callId: VGCallId,
                 withLegId legId: String,
                 andStatus status: VGLegStatus) {
    guard status == .answered, let uuid = activeCallUUID else { return }
    callProvider.reportOutgoingCall(with: uuid, connectedAt: .now)
}
Resilience

Session Error Handling

The SDK fires didReceiveSessionErrorWith(reason:) when the underlying WebSocket is disrupted during an active session. Always attempt to restore the session automatically. 

func client(_ client: VGBaseClient,
            didReceiveSessionErrorWith reason: VGSessionErrorReason) {
    print("⚠️ Session error: \(reason)")

    switch reason {
    case .tokenExpired:
        // The stored JWT is now invalid — skip it, go straight to refresh token
        refreshAndRestoreSession()

    case .pingTimeout, .transportClosed:
        // Transient network issue — try the stored JWT first
        restoreSessionIfNeeded()

    default:
        restoreSessionIfNeeded()
    }
}

private func refreshAndRestoreSession() {
    guard let refreshToken = storedRefreshToken else {
        // No refresh token — user must log in again
        clearSession()
        return
    }
    refreshJWT(refreshToken) { [weak self] result in
        switch result {
        case .success(let newJWT):
            self?.createSession(jwt: newJWT)
        case .failure:
            self?.clearSession()
        }
    }
}
Simulator

Simulator Support

PushKit and CallKit are unavailable on the iOS Simulator. Wrap all CallKit code in compile-time guards and use the SDK's WebSocket invite mode for development. 

// SDK initialisation
let config = VGClientInitConfig(loggingLevel: .info)

#if targetEnvironment(simulator)
config.enableWebsocketInvites = true   // receive invites over WebSocket — no push needed
#endif

let client = VGVoiceClient(config)
// Call answer — simulator path bypasses CallKit transactions
func answerCall(callId: String) {
#if targetEnvironment(simulator)
    client.answer(callId) { error in print("Answer: \(error?.localizedDescription ?? "OK")") }
#else
    guard let uuid = activeCallUUID else { return }
    let action = CXAnswerCallAction(call: uuid)
    callController.request(CXTransaction(action: action)) { _ in }
#endif
}
Always test the full push path on a physical device before release. The simulator cannot exercise the killed-app wakeup, session restoration under a real background budget, or CallKit lock-screen UI.
Pitfalls

Common Mistakes

1 — Calling the PushKit completion before reportNewIncomingCall

The completion block must be called only after reportNewIncomingCall has been invoked. Calling it before — or forgetting to call it at all — causes PushKit to stop delivering future pushes. Store the completion and invoke it exclusively inside reportNewIncomingCall's async callback.

2 — Skipping reportNewIncomingCall for invalid or duplicate pushes

Even if you determine the push is malformed, expired, or a duplicate, you must still call reportNewIncomingCall and immediately end the call. Skipping it breaks push delivery just as reliably as calling the completion early.

3 — Awaiting session restoration before processing the push

Chaining processCallInvitePushData after createSession delays reportNewIncomingCall by a network round-trip. In background conditions this is too slow. Fire session restoration and push processing concurrently — always.

4 — Fetching a JWT from your backend on push receipt

Adding a backend call before createSession turns the session restoration into two serial network hops. Persist the JWT at login time so the fast path needs only one hop.

5 — Hardcoding isSandbox: true

TestFlight uses production APNs. Hardcoding sandbox breaks all TestFlight push delivery. Use #if DEBUG unconditionally.

6 — Re-registering the push token on every session restore

Registering on each createSession accumulates device entries on Vonage's side and can result in duplicate pushes per call. Register only when the token actually changes (rotation) or on a fresh user login.

7 — Forgetting to unregister on logout

If you skip unregisterDeviceTokens(byDeviceId:) on logout, the device will continue receiving VoIP pushes for that user indefinitely — including after a different user logs in on the same device.

8 — Not mapping all hangup reasons to CallKit

Every call termination path — remote hangup, remote cancel, remote timeout, media failure — must call callProvider.reportCall(with:endedAt:reason:) with the correct CXCallEndedReason. Missing a path corrupts the system call log and can prevent future calls from being reported correctly.

9 — Enabling audio outside the CallKit audio callbacks

Calling VGVoiceClient.enableAudio from anywhere other than provider(_:didActivate:) causes routing issues with Bluetooth, AirPods, and CarPlay. Audio session management belongs exclusively to the CallKit delegate.

10 — Expired push certificates

VoIP certificates expire after one year. The failure mode is complete and silent — no error is surfaced anywhere in the system. Renew certificates proactively and verify push delivery immediately after uploading a new one.

Checklist

Pre-Release Checklist

  • Xcode capabilities configured Push Notifications enabled. Background Modes: Voice over IP + Background processing both checked.
  • VoIP push certificate uploaded to Vonage dashboard for both environments Sandbox certificate for debug/development. Production certificate for TestFlight and App Store.
  • isSandbox flag uses #if DEBUG — never hardcoded
  • JWT (and refresh token) persisted locally at login time Session restoration on push receipt must not require a backend call.
  • reportNewIncomingCall called for every VoIP push without exception Including invalid, duplicate, or non-Vonage pushes — report and immediately end if needed.
  • PushKit completion called only inside reportNewIncomingCall's callback Never before it, never skipped.
  • Session restoration and push processing are concurrent — not chained restoreSessionIfNeeded() and processCallInvitePushData() fire independently.
  • Token registration only on user login or token rotation Not on every createSession call.
  • unregisterDeviceTokens called on explicit user logout
  • All call termination paths call callProvider.reportCall(with:endedAt:reason:) Remote hangup, remote cancel, remote timeout, local reject, media failure.
  • Audio enabled/disabled only in didActivate / didDeactivate CallKit callbacks
  • All CallKit code guarded with #if !targetEnvironment(simulator) enableWebsocketInvites = true set for simulator builds.
  • Tested on a physical device with the app fully killed before the call The only way to exercise PushKit wakeup and the session restoration race.
  • Tested on a physical device with the app backgrounded (suspended) before the call
  • Push certificate renewal date tracked — certificates expire after 1 year
