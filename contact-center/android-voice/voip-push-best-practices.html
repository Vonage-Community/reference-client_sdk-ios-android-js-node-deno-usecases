Android VoIP Push Integration
Best Practices

A complete guide to integrating Firebase Cloud Messaging, the Android Telecom framework, and the Vonage Voice Client SDK in a production Android application — from token registration to call teardown.

Overview

Why FCM + ConnectionService?

Firebase Cloud Messaging (FCM) is the delivery mechanism for push notifications on Android. High-priority FCM messages can wake the app even in Doze mode, making them the right channel for inbound VoIP call notifications.

The Android Telecom framework (ConnectionService / TelecomManager) is the Android analogue of iOS CallKit. It integrates your VoIP app into the system's call infrastructure — providing the native incoming-call UI, system call log entries, and audio focus management. Google strongly recommends all VoIP apps use it.

The Vonage Voice Client SDK sits between the two: it decodes the FCM payload into call invites, manages the WebRTC session, and provides answer / reject / hangup operations.

Key difference from iOS

Unlike iOS PushKit, FCM imposes no strict deadline for reporting an incoming call to the Telecom framework. This means you can safely chain session restoration before processing the push payload — and the reference app does exactly that. This is the correct approach on Android.

Component responsibilities

Component Responsibility
FirebaseMessagingService Receive FCM tokens and incoming push payloads
Vonage SDK (VoiceClient) Register tokens, decode payloads, manage WebRTC session, answer/reject/hangup
TelecomHelper / TelecomManager Register the PhoneAccount, start incoming/outgoing call connections
CallConnectionService Create Connection objects; receive system UI events (answer, reject, hold)
CallConnection Represent a live call; bridge between the Telecom framework and the SDK

FCM

Firebase Cloud Messaging Integration

FirebaseMessagingService

Extend FirebaseMessagingService and override onNewToken and onMessageReceived. Keep this class thin — delegate all logic to your manager. 

class PushNotificationService : FirebaseMessagingService() {

    override fun onNewToken(token: String) {
        // Called when FCM issues a new token (first run or token rotation)
        // Store it — it will be registered with Vonage after a session is created
        App.coreContext.pushToken = token
    }

    override fun onMessageReceived(remoteMessage: RemoteMessage) {
        // All inbound FCM messages arrive here, regardless of app state
        App.coreContext.clientManager.processVoipPush(remoteMessage)
    }

    companion object {
        // Request FCM token explicitly (e.g. at login time)
        fun requestToken(onSuccess: ((String) -> Unit)? = null) {
            FirebaseMessaging.getInstance().token
                .addOnCompleteListener { task ->
                    if (task.isSuccessful) {
                        task.result?.let { token ->
                            App.coreContext.pushToken = token
                            onSuccess?.invoke(token)
                        }
                    }
                }
        }
    }
}

Registering the token with Vonage

Register the FCM token with client.registerDevicePushToken(token) immediately after a session is created. Store the returned deviceId — you will need it to unregister on logout. If the token is not yet available when the session is created, request it explicitly and register it as soon as it arrives. 

private fun registerDevicePushToken() {
    val registerCallback: (String) -> Unit = { token ->
        client.registerDevicePushToken(token) { err, deviceId ->
            err?.let {
                println("❌ Failed to register push token: $err")
            } ?: deviceId?.let {
                coreContext.deviceId = deviceId   // persist for later unregistration
                println("✅ Push token registered, deviceId: $deviceId")
            }
        }
    }

    // Use cached token if available, otherwise request one from Firebase
    coreContext.pushToken?.let { registerCallback(it) }
        ?: PushNotificationService.requestToken { registerCallback(it) }
}

fun unregisterDeviceToken(completion: () -> Unit) {
    val deviceId = coreContext.deviceId ?: run { completion(); return }

    client.unregisterDevicePushToken(deviceId) { error ->
        error?.let { println("❌ Unregister error: $it") }
        coreContext.deviceId = null
        completion()
    }
}
Token rotation: FCM can rotate tokens at any time. onNewToken is the only reliable notification. When it fires, store the token and — if a session is already active — immediately call registerDevicePushToken again. A stale token means Vonage cannot reach the device.

High-priority messages and Doze mode

Android Doze mode can delay or block FCM messages. To ensure VoIP pushes are delivered promptly even when the device is idle, Vonage's backend sends FCM messages at high priority. High-priority messages bypass Doze and immediately wake the app. You do not need to configure this in your app — it is a server-side setting in the Vonage platform.

Critical

Session Management & Credential Strategy

A Vonage session (client.createSession(jwt)) must be active before the SDK can process call invites or perform call actions. When a push arrives and the app is killed or the session has expired, restoration must happen first.

Unlike iOS — chaining is safe on Android

Android FCM has no PushKit-style deadline for showing system UI. You can safely await session restoration before calling client.processPushCallInvite. This eliminates the race condition where the user answers before a session is ready, and is the approach used in the reference app.

✗ Avoid

Process the push immediately and run session restoration concurrently. The user might answer before the session is ready, causing client.answer() to fail silently.

✓ Recommended (Android)

Restore the session first. When confirmed active, call processPushCallInvite. The session is guaranteed to exist when answer() / reject() are called.

Session creation

fun login(token: String, isUserInitiated: Boolean = true,
         onError: ((Exception) -> Unit)? = null,
         onSuccess: ((String) -> Unit)? = null) {
    if (isUserInitiated) {
        // On user-initiated login: clean up any stale device registration first
        unregisterExistingDeviceIfNeeded {
            createSession(token, onError, onSuccess)
        }
    } else {
        // Session restore: skip cleanup (same user, just reconnecting)
        createSession(token, onError, onSuccess)
    }
}

private fun createSession(token: String,
                           onError: ((Exception) -> Unit)? = null,
                           onSuccess: ((String) -> Unit)? = null) {
    client.createSession(token) { error, sessionId ->
        sessionId?.let { sid ->
            coreContext.authToken = token     // persist for future restoration
            registerDevicePushToken()        // register token now that session is active
            onSuccess?.invoke(sid)
            _sessionId.value = sid
        } ?: error?.let { onError?.invoke(it) }
    }
}

Session restoration priority chain

Priority Condition Action Typical latency
1st Active session exists (sessionId != null) Return immediately — no work needed 0 ms
2nd Stored JWT available (authToken) client.createSession(storedJWT) 100–400 ms
3rd JWT expired, refresh token available Refresh endpoint → new JWT → createSession 300–1000 ms
4th No credentials Return null — push cannot be processed 0 ms 
private fun restoreSessionIfNeeded(completion: (String?) -> Unit) {
    _sessionId.value?.let { completion(it); return }  // already active

    coreContext.authToken?.let { token ->
        restoreSessionWithToken(token, completion)
    } ?: coreContext.refreshToken?.let { refresh ->
        restoreSessionWithRefreshToken(refresh, completion)
    } ?: completion(null)   // no credentials
}
Persist the JWT at login time. If createSession requires a live backend call to obtain a JWT every time a push arrives, you are adding an extra network hop before the session is ready. On slow connections this compounds the restoration latency and degrades call answer reliability.

Preventing token accumulation on re-login

On a fresh user-initiated login, always unregister the previously stored device before creating a new session. Skipping this causes Vonage to accumulate multiple device registrations, which leads to duplicate push notifications for the same call. 

private fun unregisterExistingDeviceIfNeeded(completion: () -> Unit) {
    val existingDeviceId = coreContext.deviceId ?: run { completion(); return }
    val authToken = coreContext.authToken ?: run {
        coreContext.deviceId = null; completion(); return
    }
    // Create a temporary session just to perform the unregistration, then tear it down
    client.createSession(authToken) { _, sessionId ->
        sessionId ?: run { coreContext.deviceId = null; completion(); return@createSession }
        unregisterDeviceToken {
            client.deleteSession { completion() }
        }
    }
}
Core

Handling an Incoming VoIP Push

The full path from FCM delivery to the system incoming-call UI:

FCM delivers message — onMessageReceived fires
The system invokes PushNotificationService.onMessageReceived. Unlike iOS PushKit, there is no strict deadline — the app can perform async work before showing the call UI.
App delegates to VoiceClientManager.processVoipPush
The service stays thin. All logic lives in the manager.
VoiceClientManager restores session (awaited)
restoreSessionIfNeeded is called and its completion is awaited. Processing only continues once a sessionId is confirmed. If restoration fails, the push is dropped — the call cannot be answered without a session.
Vonage SDK client.processPushCallInvite(data)
Validates the payload type via VoiceClient.getPushNotificationType. If it's an INCOMING_CALL, decodes it and fires the setCallInviteListener callback.
Vonage SDK setCallInviteListener fires
Callback delivers callId, from, and channel type. The app creates its call state and forwards to the Telecom framework.
Telecom TelecomManager.addNewIncomingCall
Triggers CallConnectionService.onCreateIncomingConnection. A CallConnection is created, set to ringing, and attached to CoreContext.activeCall.
System shows native incoming call UI
The OS displays the full-screen incoming call notification or heads-up notification depending on device state and lock screen. The user can answer, decline, or ignore.

Implementation

// VoiceClientManager.kt
fun processVoipPush(remoteMessage: RemoteMessage) {
    // Step 1: ensure a session exists before touching the push payload
    restoreSessionIfNeeded { sessionId ->
        sessionId ?: run {
            println("❌ Session restoration failed — cannot process push")
            return@restoreSessionIfNeeded
        }
        // Step 2: now safe to process — session is guaranteed active
        processIncomingPush(remoteMessage)
    }
}

fun processIncomingPush(remoteMessage: RemoteMessage) {
    val dataString = remoteMessage.data.toString()
    val type = VoiceClient.getPushNotificationType(dataString)
    if (type == PushType.INCOMING_CALL) {
        // Triggers setCallInviteListener callback
        client.processPushCallInvite(dataString)
    }
}

// In setClientListeners()
client.setCallInviteListener { callId, from, type ->
    // Reject if another call is already active
    coreContext.activeCall.value?.let { return@setCallInviteListener }
    placeIncomingCall(callId, from, type)
    startForegroundService(context)   // required for background audio recording
}

private fun placeIncomingCall(callId: CallId, caller: String, type: VoiceChannelType) {
    try {
        coreContext.telecomHelper.startIncomingCall(callId, caller, type)
        navigateToMainActivity(context)   // bring app to foreground
    } catch (e: Exception) {
        // Telecom permission denied or phone account not enabled
        client.reject(callId) {}
    }
}
Telecom

Telecom Framework Integration

PhoneAccount registration

A PhoneAccount must be registered with TelecomManager before any call can be placed or received. Registration is idempotent — do it on every app start. The user must enable the account in system settings the first time; if it is not enabled, prompt them with the system enable-account activity. 

class TelecomHelper(private val context: Context) {
    private val telecomManager = context.getSystemService(Context.TELECOM_SERVICE) as TelecomManager
    private val phoneAccountHandle: PhoneAccountHandle

    init {
        val componentName = ComponentName(context, CallConnectionService::class.java)
        phoneAccountHandle = PhoneAccountHandle(componentName, "YourAppVoIP")
        registerPhoneAccount()
    }

    private fun registerPhoneAccount() {
        telecomManager.getPhoneAccount(phoneAccountHandle)
            ?: PhoneAccount
                .builder(phoneAccountHandle, "Your App VoIP Calling")
                .setCapabilities(PhoneAccount.CAPABILITY_CALL_PROVIDER)
                .build()
                .also { telecomManager.registerPhoneAccount(it) }

        if (!telecomManager.getPhoneAccount(phoneAccountHandle).isEnabled) {
            promptUserToEnableAccount()
        }
    }
}
Samsung and some OEM devices have known issues where ConnectionService.onCreateOutgoingConnection is never called. Implement a fallback timer (e.g. 3 s) that mocks the outgoing connection directly if the ConnectionService does not respond, allowing the call to proceed without the Telecom framework.

Starting an incoming call

fun startIncomingCall(callId: CallId, from: String, type: VoiceChannelType) {
    val extras = Bundle()
    extras.putString("EXTRA_CALL_ID", callId)
    extras.putString("EXTRA_FROM", from)
    // This triggers CallConnectionService.onCreateIncomingConnection
    telecomManager.addNewIncomingCall(phoneAccountHandle, extras)
}

CallConnectionService

Extend ConnectionService to create Connection objects for both incoming and outgoing calls. Extract the call metadata from the Bundle passed via extras. 

class CallConnectionService : ConnectionService() {

    override fun onCreateIncomingConnection(
        phoneAccount: PhoneAccountHandle?, request: ConnectionRequest?
    ): Connection {
        val extras = request!!.extras
        val callId = extras.getString("EXTRA_CALL_ID")!!
        val from   = extras.getString("EXTRA_FROM")

        return CallConnection(callId).apply {
            setAddress(Uri.parse(from), TelecomManager.PRESENTATION_ALLOWED)
            setCallerDisplayName(from, TelecomManager.PRESENTATION_ALLOWED)
            setRinging()   // shows system incoming-call UI
        }
    }

    override fun onCreateOutgoingConnection(
        phoneAccount: PhoneAccountHandle?, request: ConnectionRequest?
    ): Connection {
        val extras = request!!.extras
        val callId = extras.getString("EXTRA_CALL_ID")!!
        val to     = extras.getString("EXTRA_TO")

        return CallConnection(callId).apply {
            setAddress(Uri.parse(to), TelecomManager.PRESENTATION_ALLOWED)
            setCallerDisplayName(to, TelecomManager.PRESENTATION_ALLOWED)
            setDialing()
        }
    }
}

CallConnection — user action callbacks

Override Connection callbacks to bridge system UI actions to the Vonage SDK. Keep the connection class decoupled from business logic — delegate everything to your manager. 

class CallConnection(val callId: CallId) : Connection() {
    private val clientManager = App.coreContext.clientManager

    init {
        connectionProperties  = connectionProperties or PROPERTY_SELF_MANAGED
        connectionCapabilities = connectionCapabilities or
            CAPABILITY_MUTE or CAPABILITY_SUPPORT_HOLD or CAPABILITY_HOLD
        audioModeIsVoip = true
    }

    // User tapped Answer on the system UI
    override fun onAnswer() { clientManager.answerCall(this) }

    // User tapped Decline on the system UI
    override fun onReject() { clientManager.rejectCall(this) }

    // User tapped End during an active call
    override fun onDisconnect() { clientManager.hangupCall(this) }

    // Mute toggled from system UI (e.g. Android Auto, wear OS)
    override fun onMuteStateChanged(isMuted: Boolean) {
        if (isMuted) clientManager.muteCall(this)
        else clientManager.unmuteCall(this)
    }

    // Hold toggled from system UI
    override fun onHold()   { clientManager.holdCall(this) }
    override fun onUnhold() { clientManager.unholdCall(this) }

    // DTMF from system dialer
    override fun onPlayDtmfTone(c: Char) { clientManager.sendDtmf(this, c.toString()) }
}
SDK Listeners

Vonage SDK Event Listeners

Set all listeners once, immediately after initialising VoiceClient, before creating any session. Listeners fire on a background thread — update UI state on the main thread.

Hangup

client.setOnCallHangupListener { callId, quality, reason ->
    takeIfActive(callId)?.apply {
        val cause = when (reason) {
            HangupReason.remoteReject           -> DisconnectCause.REJECTED
            HangupReason.remoteHangup           -> DisconnectCause.REMOTE
            HangupReason.localHangup             -> DisconnectCause.LOCAL
            HangupReason.mediaTimeout            -> DisconnectCause.BUSY
            HangupReason.remoteNoAnswerTimeout   -> DisconnectCause.CANCELED
        }
        cleanUp(DisconnectCause(cause))
    }
}

Invite cancelled by remote

client.setCallInviteCancelListener { callId, reason ->
    takeIfActive(callId)?.apply {
        val cause = when (reason) {
            VoiceInviteCancelReason.AnsweredElsewhere -> DisconnectCause(DisconnectCause.ANSWERED_ELSEWHERE)
            VoiceInviteCancelReason.RejectedElsewhere -> DisconnectCause(DisconnectCause.REJECTED)
            VoiceInviteCancelReason.RemoteCancel      -> DisconnectCause(DisconnectCause.CANCELED)
            VoiceInviteCancelReason.RemoteTimeout     -> DisconnectCause(DisconnectCause.MISSED)
        }
        cleanUp(cause)
    } ?: stopForegroundService(context)
}

Media events

client.setOnCallMediaDisconnectListener { callId, reason ->
    takeIfActive(callId)?.cleanUp(DisconnectCause(DisconnectCause.ERROR))
}

client.setOnCallMediaReconnectingListener { callId ->
    takeIfActive(callId)?.setInitializing()   // show reconnecting UI
}

client.setOnCallMediaReconnectionListener { callId ->
    takeIfActive(callId)?.setActive()           // resume active UI
}

Leg status (outbound call answered by remote)

client.setOnLegStatusUpdate { callId, legId, status ->
    takeIfActive(callId)?.apply {
        if (status == LegStatus.answered) {
            setActive()   // transition connection to active state
        }
    }
}

Call operations

fun answerCall(call: CallConnection) {
    client.answer(call.callId) { err ->
        if (err != null) call.cleanUp(DisconnectCause(DisconnectCause.ERROR))
        else call.setActive()
    }
}

fun rejectCall(call: CallConnection) {
    client.reject(call.callId) { err ->
        val cause = if (err != null) DisconnectCause.ERROR else DisconnectCause.REJECTED
        call.cleanUp(DisconnectCause(cause))
    }
}

fun hangupCall(call: CallConnection) {
    client.hangup(call.callId) { err ->
        // On error: hangup listener won't fire, so clean up manually
        err?.let { call.cleanUp(DisconnectCause(DisconnectCause.LOCAL)) }
        // On success: setOnCallHangupListener handles cleanup
    }
}

fun holdCall(call: CallConnection) {
    client.enableEarmuff(call.callId) { err ->
        err ?: client.mute(call.callId) { err2 ->
            err2 ?: call.toggleHold()   // updates state + calls setOnHold()
        }
    }
}

fun sendDtmf(call: CallConnection, digit: String) {
    client.sendDTMF(call.callId, digit) { err ->
        err?.let { println("DTMF error: $it") }
    }
}
Outbound

Outbound Calls

For outbound calls, start the SDK call first, then notify the Telecom framework. The framework creates the CallConnection via onCreateOutgoingConnection. 

fun startOutboundCall(callContext: Map<String, String>? = null) {
    client.serverCall(callContext) { err, callId ->
        err?.let { println("Outbound call error: $it"); return@serverCall }
        callId?.let { id ->
            val callee = callContext?.get("callee") ?: "unknown"
            placeOutgoingCall(id, callee)
            startForegroundService(context)
        }
    }
}

private fun placeOutgoingCall(callId: CallId, to: String) {
    try {
        val extras = Bundle().apply {
            putString("EXTRA_TO", to)
            putString("EXTRA_CALL_ID", callId)
            putParcelable(TelecomManager.EXTRA_PHONE_ACCOUNT_HANDLE, phoneAccountHandle)
        }
        telecomManager.placeCall(Uri.parse("tel:$to"), extras)

        // Fallback timer: some OEMs never call onCreateOutgoingConnection
        TimerManager.startTimer(3000) { mockOutgoingConnection(callId, to) }

    } catch (e: Exception) {
        // Telecom framework unavailable — abort the call via SDK
        client.hangup(callId) {}
    }
}
Android Requirement

Foreground Service for Audio

Since Android 14 (API 34), a foreground service of type microphone is required to record audio, even when the app is in the foreground. For API 26–33 it is required when the app is in the background. Start the service when a call is initiated and stop it when the call ends. 

// Declare in AndroidManifest.xml
<service
    android:name=".services.AudioRecorderService"
    android:exported="false"
    android:foregroundServiceType="microphone"/>

<uses-permission android:name="android.permission.FOREGROUND_SERVICE"/>
<uses-permission android:name="android.permission.FOREGROUND_SERVICE_MICROPHONE"/>
// Start when call begins (inbound invite or outbound server call)
fun startForegroundService(context: Context) {
    val intent = Intent(context, AudioRecorderService::class.java)
    context.startForegroundService(intent)
}

// Stop when call ends (every termination path)
fun stopForegroundService(context: Context) {
    val intent = Intent(context, AudioRecorderService::class.java)
    context.stopService(intent)
}
The foreground service must be started before or immediately when the call invite fires. If it is started too late — after the audio session has already been requested — some devices will deny microphone access while backgrounded.
Resilience

Session Error Handling

The SDK fires setSessionErrorListener when the WebSocket drops during an active session. Attempt restoration automatically, using the token expiry signal to choose the right strategy. 

client.setSessionErrorListener { reason ->
    when (reason) {
        SessionErrorReason.TokenExpired ->
            // Stored JWT is invalid — skip it, go straight to refresh token
            attemptSessionRestoration(skipAuthToken = true)

        SessionErrorReason.TransportClosed,
        SessionErrorReason.PingTimeout ->
            // Transient network issue — try stored JWT first
            attemptSessionRestoration(skipAuthToken = false)
    }
}

private fun attemptSessionRestoration(skipAuthToken: Boolean = false) {
    val handleFailure = {
        clearSession()
        // Prompt user to log in again
    }
    val fallbackToRefresh = {
        coreContext.refreshToken?.let { token ->
            restoreSessionWithRefreshToken(token) { it ?: handleFailure() }
        } ?: handleFailure()
    }

    if (!skipAuthToken) {
        coreContext.authToken?.let { token ->
            restoreSessionWithToken(token) { it ?: fallbackToRefresh() }
        } ?: fallbackToRefresh()
    } else {
        fallbackToRefresh()
    }
}
Pitfalls

Common Mistakes

1 — Processing the push before session restoration

On Android, unlike iOS, you should chain session restoration before calling processPushCallInvite. Processing the push without an active session may silently fail, or the call invite listener may not fire, resulting in a missed call with no UI shown.

2 — Fetching the JWT from your backend on every push

If session restoration requires a live backend call before createSession, you double the latency on every cold-start push. Persist the JWT at login time so restoration needs only one createSession call.

3 — Not registering the push token after every session creation

Call client.registerDevicePushToken inside the createSession success callback — not once at app launch. The session must exist before registration. If you miss this, the device will not receive VoIP pushes until the next re-login.

4 — Not unregistering on logout

Skipping client.unregisterDevicePushToken at logout leaves the device registered on Vonage's servers. After a different user logs in, the old user's incoming calls will still arrive as pushes on this device.

5 — Registering multiple device tokens

Calling registerDevicePushToken on every createSession call, regardless of whether the token has changed, accumulates duplicate registrations. This causes Vonage to deliver multiple pushes for the same call. Before re-registering, compare the current token with the stored one; skip registration if they are equal.

6 — Stale FCM token not re-registered

FCM can rotate tokens without warning. onNewToken is the only callback for this event. When it fires with a new token, immediately store it and call registerDevicePushToken if a session is active. Failing to handle token rotation causes all future pushes to be silently undeliverable.

7 — Missing foreground service declaration or wrong type

On API 34+ the foreground service must be declared with foregroundServiceType="microphone". Omitting the type, or declaring it only at runtime, causes the system to deny audio recording when the app is not in the foreground.

8 — Not handling ConnectionService failures on Samsung/OEM devices

Some Samsung, Xiaomi, and other OEM devices do not reliably invoke onCreateOutgoingConnection. Without a fallback timer that creates the connection directly, outbound calls silently fail on these devices. Always implement a timeout fallback for outgoing connections.

9 — PhoneAccount not enabled

The user must explicitly enable your app's PhoneAccount in system settings on first run. If the account is not enabled, every call to addNewIncomingCall or placeCall will throw. Check account status before placing calls and redirect the user to the system enable-account activity if needed.

10 — Not handling all hangup/cancel reasons

Every call termination path — remote hangup, remote cancel, remote timeout, media disconnect, local error — must call connection.disconnect(disconnectCause) followed by connection.destroy() (or the equivalent cleanUp helper). Missing a path leaves zombie Connection objects in the Telecom framework, breaking future calls.

11 — Re-creating VoiceClient on each push

Initialise VoiceClient once in your Application.onCreate() or a singleton and reuse it throughout the app lifecycle. Recreating it clears all listeners and loses session state.

Checklist

Pre-Release Checklist

  • google-services.json present in app/ and excluded from version control
  • All required permissions declared in AndroidManifest.xml RECORD_AUDIO, MANAGE_OWN_CALLS, CALL_PHONE, FOREGROUND_SERVICE, FOREGROUND_SERVICE_MICROPHONE, READ_PHONE_STATE.
  • RECORD_AUDIO runtime permission requested and granted before any call attempt
  • PushNotificationService, AudioRecorderService, and CallConnectionService all declared in manifest AudioRecorderService must include foregroundServiceType="microphone".
  • VoiceClient initialised once (Application.onCreate or singleton) with all listeners set before first session
  • Session restoration awaited before processPushCallInvite Android has no PushKit deadline — chaining is safe and recommended.
  • JWT (and refresh token) persisted locally at login time Session restoration on push receipt should not require a live backend call.
  • registerDevicePushToken called inside createSession success callback Not at app launch and not on every session restore — only when token or session changes.
  • unregisterDevicePushToken called on explicit user logout
  • FCM token rotation handled in onNewToken Store updated token and re-register if a session is already active.
  • PhoneAccount registered on app start; user prompted to enable it if disabled
  • OEM ConnectionService fallback timer implemented for outgoing calls Handles Samsung and other devices that don't reliably invoke onCreateOutgoingConnection.
  • Foreground service started when call begins and stopped on every termination path
  • All hangup/cancel reasons mapped to the correct DisconnectCause
  • Session error handling restores session automatically; token-expired skips stored JWT
  • Tested on a physical device with app fully killed before the inbound call The only way to exercise FCM wakeup and session restoration from scratch.
  • Tested on at least one Samsung or other OEM device (not just Pixel/emulator) ConnectionService behaviour varies significantly across manufacturers.
